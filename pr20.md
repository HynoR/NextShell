# PR #20 审查结果

## 结论

当前不建议合并。该 PR 在 AI 执行链路上有 4 个需要提交者自行修复的问题，其中 3 个为 P1，1 个为 P2。

## 问题列表

### 1. P1: AI 命令可能执行到和当前终端不一致的主机

- 位置: `apps/desktop/src/renderer/components/WorkspaceLayout.tsx`
- 问题: `AiChatPane` 传入的是当前终端 `sessionId`，但 `connectionId` 用的是树上选中的 `activeConnectionId`。当终端聚焦在机器 A、左侧树切到机器 B 时，AI 审批后的命令会真正执行在 B 上，但界面会把命令回显到 A 的终端里，存在明显误导和误操作风险。
- 简短修复:
  - 给 `AiChatPane` 传 `activeTerminalConnection?.id`，不要传树选中的 `activeConnectionId`。
  - 复用已有对话时，同步刷新对话绑定的 `sessionId` / `connectionId`，避免对话长期绑定旧机器。

### 2. P1: `AiAbort` 无法中止已批准的执行计划

- 位置: `apps/desktop/src/main/services/ai/ai-service.ts`
- 问题: 当前 `abort()` 只会取消聊天流式生成阶段创建的 `AbortController`。批准计划后的 `executePlan()` 和执行后的分析流都没有注册取消控制，也没有轮询取消状态。结果是 UI 侧就算显示“已停止”，后台命令仍可能继续执行。
- 简短修复:
  - 为“计划执行阶段”和“执行后分析阶段”分别接入可取消控制。
  - `AiAbort` 需要真正中断远端执行循环，并向前端回传明确的 `aborted` / `failed` 状态。
  - 前端停止按钮不要只绑定聊天流式阶段，要覆盖执行阶段。

### 3. P1: 某一步命令失败后，后续步骤仍会继续执行

- 位置: `apps/desktop/src/main/services/ai/ai-service.ts`
- 问题: 当前逻辑在 `exitCode !== 0` 时只把该步骤标记为 `failed`，但不会终止循环，后续步骤仍会继续执行。对 AI 生成的运维计划来说，这会让后续命令运行在错误前提上，风险很高。
- 简短修复:
  - 对非零退出码默认立即停止整个计划。
  - 如果未来要支持“失败后继续”，需要在计划 schema 里显式表达，而不是默认继续。

### 4. P2: 多步 shell 计划不会共享 shell 上下文

- 位置: `apps/desktop/src/main/services/ai/ai-service.ts`
- 问题: 只要存在 `connectionId`，每一步都会走 `execCommand()`，而 `execCommand()` 底层是新的 `connection.exec(...)` 调用，不是复用当前交互式 shell。这样像 `cd`、`export`、临时变量、shell 选项等上下文都不会保留，多步计划和用户审批时理解的不一致。
- 简短修复:
  - 如果设计目标是“按终端会话执行计划”，就应在绑定的终端会话内串行执行。
  - 如果设计目标是“无状态命令执行”，就必须收紧 prompt 和 plan schema，明确要求每一步都是独立、可单独执行的命令。

## 建议给提交者的修复顺序

1. 先修复主机绑定错误，避免命令打到错误机器。
2. 再补全执行中止能力，确保危险命令可停。
3. 然后修复失败即停逻辑，避免错误链式扩散。
4. 最后统一“执行模型”定义，明确是交互式 shell 还是无状态 exec。
